- [x] `hls_manager.py`: segment_key values are embedded in M3U8 playlist lines without escaping вЂ” a key containing `\n` or `#` corrupts the playlist and breaks playback for the affected job.
  - Plan: add a dedicated segment-path sanitizer (e.g., `_sanitize_segment_uri_path`) that strips CR/LF, rejects leading `#`, and percent-encodes unsafe URI characters before writing playlist lines.
  - Plan: use the sanitizer in both `generate_media_playlist()` and `_generate_subtitle_playlist()` when appending `/segment/{job_id}/{segment_key}` URIs.

- [x] `hls_manager.py`: subtitle playlists emit `#EXTINF:0.000,` when job duration is NULL or 0, which is invalid per the HLS spec and causes players to reject or skip the subtitle track.
  - Plan: compute `subtitle_duration = max(job["duration"] or 0, Config.HLS_SEGMENT_DURATION, 0.001)` and use it for both `#EXTINF` and target-duration derivation.

  - Plan: centralize fallback bitrate logic in a helper that never divides by 0 and never returns raw `file_size * 8` as a final BANDWIDTH value.
  - Plan: when duration is missing/zero, estimate bitrate from configured segment duration (or clamp to a sane floor/ceiling) so values stay realistic for ABR selection.
  - Plan: add tests for zero/NULL duration in both video-track and legacy single-stream master playlist paths, asserting BANDWIDTH remains within expected range and is always positive.
- [x] `app.py`: the global `_aiohttp_session` is recreated without a lock вЂ” multiple concurrent coroutines can each create a new `ClientSession`, leaking the earlier sessions as open sockets until the OS reclaims them, eventually exhausting the connection pool.
- [x] `app.py`: temp files created by `tempfile.mkstemp()` inside the segment download path are not reliably cleaned up when the download task is cancelled or times out, leading to gradual disk exhaustion.
- [x] `stream_analyzer.py`: `stream["index"]` uses bare dict access; if ffprobe omits the `index` field for any stream object (seen with some containers), an unhandled `KeyError` crashes the analysis stage and permanently fails the job.

## P2 вЂ” Reliability

- [ ] `app.py:406вЂ“420,2010вЂ“2015`: `_aiohttp_session` recreated without a lock and outside the async event loop вЂ” two concurrent Flask threads can both see the session as `None`/`closed`, each call `aiohttp.ClientSession()`, and only one assignment survives while the other leaks open sockets; creating a `ClientSession` from a sync Flask thread is also incorrect per aiohttp's event-loop affinity requirement. Fix: schedule session creation via `_run_async()` inside the persistent async loop and protect the recreation check with a `threading.Lock` so only one coroutine ever creates the session.
- [ ] `app.py:1447вЂ“1485`: upload finalization leaks the assembled temp file when `_queue_local_file()` raises вЂ” `_remove_pending_upload()` is called at line 1468 before `_queue_local_file()`, so if queuing throws, the `upload_id` is gone from tracking but the file in `uploads/` is never deleted and the per-IP counter has already been decremented, leaving orphaned disk usage the cleanup task will not find. Fix: wrap `_queue_local_file()` in a try/except that deletes the temp file and re-raises, or move `_remove_pending_upload()` to after a successful enqueue.
- [ ] `app.py:162вЂ“175`: `_request_job_stop()` calls `future.cancel()` on in-flight Telegram upload futures but never handles the case where `cancel()` returns `False` (future already running) вЂ” those uploads continue in the background, pushing segments to Telegram that are never registered in the DB and become permanently orphaned in the channel. Fix: after cancellation, await remaining futures with a short timeout inside the async loop so the uploader's cancel-event check can terminate them cleanly before the job is marked finished.
- [ ] `app.py:1275вЂ“1343`: watch-folder polling crashes the entire watch thread when a file disappears between directory scan and stability check вЂ” `_watch_file_signature()` calls `os.stat()` with no exception handling, so a `FileNotFoundError` or `PermissionError` propagates through `_claim_watch_file_if_stable()` into the top-level poll loop and kills auto-ingest until the app is restarted. Fix: wrap `os.stat()` in a try/except returning `None` on any OS error, and treat a `None` signature in `_claim_watch_file_if_stable()` as "file gone вЂ” skip".
- [ ] `telegram_uploader.py:169вЂ“199,320вЂ“340`: `reload_bots()` replaces `self.bots` and nulls `self._bot_locks` without a lock вЂ” a concurrent upload task that already read a `bot_index` against the old list length will index into the newly built (possibly shorter) `_bot_locks`, raising `IndexError`; two tasks that both see `_bot_locks is None` will each allocate a new lock list and the second write discards the first, letting two uploads proceed to the same bot concurrently and defeating per-bot serialisation. Fix: protect `self.bots`, `self._bot_counter`, and `self._bot_locks` under a single `threading.Lock` held across `reload_bots()`, `_next_bot()`, and the lazy `_bot_locks` initialisation.
- [ ] `video_processor.py:907вЂ“933`: when one ABR tier encode fails, `cancel_event.set()` is called but already-running `ThreadPoolExecutor` workers cannot be interrupted (`future.cancel()` returns `False`), so those threads continue writing `.ts` files into `processing/<job_id>/` alongside any already-complete tiers; if `cleanup()` then races against a still-writing thread, `shutil.rmtree()` can fail or leave a partial directory that contaminates a subsequent run. Fix: after the `ThreadPoolExecutor` context exits (all workers done or timed out), unconditionally invoke `cleanup(job_id)` before re-raising so partial output is removed in the failure path.
- [ ] `video_processor.py:1022вЂ“1027`: `cleanup()` calls `shutil.rmtree()` with no exception handling вЂ” if any file in the processing directory is still held open by a lingering FFmpeg subprocess or encode thread, the `rmtree` raises and the exception propagates into the caller's `finally` block, silently replacing the original job-failure exception and making root-cause diagnosis impossible. Fix: wrap `shutil.rmtree()` in a try/except, log a warning on failure, and ensure the function always returns without raising so it is safe to call from `finally` blocks.
- [ ] `database.py:33вЂ“65`: thread-local SQLite connections have no staleness detection вЂ” if the DB file is moved or a WAL-recovery event invalidates the cached handle, the stale `_local.conn` passes the `if not hasattr(вЂ¦) or _local.conn is None` guard and all subsequent queries on that thread raise `sqlite3.OperationalError` with no automatic reconnection; `_handle_corrupt_db()` calls `_reset_conn()` only for the calling thread, leaving all other threads' connections in an unknown broken state. Fix: in `_get_conn()`, catch `sqlite3.OperationalError`, call `_reset_conn()`, and retry the connection once before re-raising so threads self-heal after transient DB file disruptions.
- [ ] `app.py:1210вЂ“1221`: zero-count entries in `_pending_uploads_per_ip` (a `defaultdict`) are never removed вЂ” over a long-running session the dict accumulates one entry per distinct client IP that has ever submitted an upload, growing without bound and inflating memory usage on busy instances. Fix: after decrementing, add `if _pending_uploads_per_ip[ip] == 0: del _pending_uploads_per_ip[ip]` to remove exhausted entries immediately.

## P3 вЂ” Data Model

- [ ] `database.py:208` вЂ” `jobs.filename` has no NOT NULL constraint, but every code path that reads jobs (`database.py:575`, UI) expects a non-null filename. A NULL here breaks job-list queries silently. Fix: schema migration (revision 7) to add `NOT NULL` via table-recreate (SQLite constraint-alter workaround).
- [ ] `database.py:221` вЂ” `tracks.track_type` accepts any TEXT but code branches exclusively on `'video'`, `'audio'`, `'subtitle'` (`database.py:518-519`, `633-634`). An invalid value silently breaks playlist generation. Fix: add `CHECK(track_type IN ('video', 'audio', 'subtitle'))` in schema migration.
- [ ] `database.py:260` вЂ” `jobs.media_type` has no CHECK constraint, but `database.py:579-585` filters by hard-coded category strings. Arbitrary values from `PATCH /api/jobs/<job_id>` silently persist. Fix: add a CHECK constraint listing valid media types, or validate input in `app.py` before the DB write.
- [ ] `database.py:237` вЂ” `segments.segment_key` is only constrained by `UNIQUE(job_id, segment_key)`, but the `prefix/filename` format is assumed everywhere (split on `/` in `app.py:1947`, `hls_manager.py:278`). A key missing a `/` corrupts M3U8 output. Fix: add `CHECK(segment_key GLOB '*/*')` in schema migration, or enforce the format before every insert.
- [ ] `database.py:255,265` вЂ” `segments.duration` defaults to `0`, which is indistinguishable from an actual 0-second segment. `hls_manager.py:265-266` uses 0 as a sentinel meaning "unknown, use fallback", causing `#EXTINF:0.000,` in playlists (invalid per HLS spec, also covered at P0 level for subtitle playlists). - [ ] `database.py:262,268` вЂ” Integer-boolean columns `has_thumbnail` and `is_series` lack `CHECK(col IN (0,1))` constraints. SQLite accepts any integer, so a stale migration or a code bug could store `2` or `-1`; `bool()` coercion in Python would still be truthy but comparison to `== 1` would silently fail. - [ ] `database.py:268-271` вЂ” `is_series`, `season_number`, `episode_number`, `part_number` are all nullable with no cross-column constraint. - [ ] `database.py:235` вЂ” `segments.bot_index` is an unvalidated INTEGER with no foreign-key or range constraint. - [ ] `database.py:211` вЂ” `jobs.video_codec` is nullable, but `hls_manager.py:158` relies on it to construct the `CODECS=` M3U8 attribute. A NULL produces an invalid or missing CODECS string. Fix: default to `'unknown'` at insert or add `NOT NULL DEFAULT 'unknown'` in migration.
- [ ] `database.py:223,249` вЂ” `tracks.codec` and `tracks.bitrate` are nullable, but `hls_manager.py:134,141` calls `_parse_bitrate(t["bitrate"])` without a NULL guard, and `hls_manager.py:158` builds codec strings from `t["codec"]`. - [ ] `database.py:286` вЂ” `bots.channel_id` is declared NOT NULL but has no `CHECK(channel_id < 0)` constraint. Telegram channel IDs must be negative; a positive value is accepted by the DB but fails at runtime during uploads. Fix: add the CHECK constraint in migration.
- [ ] `database.py:241` вЂ” Missing index on `jobs(media_type)`. Every call to `list_jobs()` with a category filter does a full `jobs` table scan. Fix: schema migration adding `CREATE INDEX IF NOT EXISTS idx_jobs_media_type ON jobs(media_type)`.
- [ ] `database.py:215` вЂ” No index on `jobs(created_at)` despite `list_jobs()` sorting by `created_at DESC` on every request. Fix: schema migration adding `CREATE INDEX IF NOT EXISTS idx_jobs_created_at ON jobs(created_at DESC)`.
- [ ] `database.py:250,427,440` вЂ” `tracks.original_stream_index` defaults to `-1` as a "missing stream" sentinel but has no `CHECK(original_stream_index >= -1)` constraint. Fix: add CHECK in migration and document the -1 convention in a code comment.
- [ ] `config.py` (bot loading) вЂ” `Config.BOTS` is built from `TELEGRAM_BOT_TOKEN_*` env vars with no deduplication check. Accidentally duplicated tokens are silently accepted and inflate the round-robin pool, causing two entries to upload to the same bot in parallel and bypassing per-bot serialisation in `telegram_uploader.py`. - [ ] `database.py:278` вЂ” `settings.key` accepts arbitrary TEXT. ## P4 вЂ” Security Hardening

- [ ] `app.py:178-184` вЂ” X-Forwarded-For header is trusted unconditionally when `BEHIND_PROXY=true`; any client can spoof this header to present an arbitrary source IP and bypass per-IP rate limiting (`UPLOAD_RATE_LIMIT_MAX_REQUESTS`) and `MAX_PENDING_UPLOADS_PER_IP`. Fix: validate that requests arrive from a configured trusted-proxy CIDR before accepting the forwarded IP, or document that `BEHIND_PROXY` must only be set when the app is behind a controlled reverse-proxy that strips the header on ingress.
- [ ] `app.py:1361-1370` вЂ” `_build_done_destination()` guards against `..` at the start of `rel_path` but does not resolve symlinks before path construction; a symlinked file or directory inside `WATCH_ROOT` can produce a destination outside `WATCH_DONE_DIR`, enabling arbitrary file placement. Fix: call `os.path.realpath()` on both `root` and `path` before computing `rel_path`, and verify the resolved destination starts with the realpath of `done_dir`.
- [ ] `app.py:237-243` вЂ” `_is_origin_allowed()` performs exact string comparison against configured origins without parsing them as URIs or enforcing a scheme; an entry like `example.com` (no scheme) is not blocked for `https://example.com.evil.com`. Fix: parse each configured origin as a `urllib.parse.urlsplit()` result, compare scheme + netloc exactly, and reject malformed entries at config-load time.
- [ ] `app.py:926,973` вЂ” `/api/bots/health` and `/api/bots/add` return `str(exc)` verbatim in JSON error responses, which can expose Telegram bot token fragments, internal file paths, or Python stack details to any caller. Fix: log the full exception server-side and return a generic message (e.g. `"Health check failed"` / `"Bot validation failed"`) to the client.
- [ ] `app.py:868` вЂ” The settings API accepts arbitrary strings for FFmpeg-related parameters (`VIDEO_BITRATE`, `AUDIO_BITRATE`, `PREFERRED_ENCODER`, `VAAPI_DEVICE`) without format validation; `VAAPI_DEVICE` is passed verbatim as an argument to FFmpeg, and a malformed value could redirect device access or cause unexpected encoder behaviour. Fix: validate `VIDEO_BITRATE` and `AUDIO_BITRATE` against the existing `_BITRATE_RE` regex; restrict `PREFERRED_ENCODER` to the known set `{vaapi, nvenc, qsv, cpu}`; validate `VAAPI_DEVICE` against the `/dev/dri/renderD*` pattern (or a user-configured allowlist).
- [ ] `app.py` (HLS + segment routes) вЂ” `job_id` URL parameters accepted by `/hls/<job_id>/вЂ¦` and `/segment/<job_id>/вЂ¦` are passed to SQLite queries with no format pre-check; while parameterised queries prevent SQL injection, arbitrary-length or specially crafted strings reach the DB layer unnecessarily. Fix: validate `job_id` against a UUID4 regex at the top of each route handler and return 400 immediately on mismatch, reducing attack surface and easing log triage.
- [ ] `database.py:117` вЂ” `_list_table_columns()` interpolates its `table_name` argument directly into a PRAGMA statement (`f"PRAGMA table_info({table_name})"`); SQLite does not support parameter binding for PRAGMA, so if this helper were ever called with external input the table name could be used to inject additional SQL. Fix: add an explicit allowlist check (`if table_name not in {"jobs", "tracks", "segments", "settings", "bots", "schema_migrations"}: raise ValueError`) before the execute call.
- [ ] `database.py:600-692` вЂ” `list_jobs()` and `count_jobs()` assemble SQL query strings using f-strings (conditional GROUP BY clauses, series count subqueries, WHERE concatenation); although `group_by` is validated against a whitelist, the fragile string-construction pattern makes it easy to introduce injection vectors when extending the query in future. Fix: replace f-string query assembly with explicit per-branch query constants so each variant is a static string with only parameterised placeholders.
